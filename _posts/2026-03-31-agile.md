## 1. Agile
> 빠르게 만들고, 빠르게 피드백 받고, 빠르게 개선하는 프로세스 모델

**Agile 프로세스의 주요 가치**
- 형식적인 문서보다는 커뮤니케이션
- 사용자는 문서가 아니라 실행하는 소프트웨어를 통하여 요구 확인
- 사용자의 요구는 프로젝트 중간에 바뀔 수 있음을 고려
- 짧은 주기동안 요구정의, 구현, 테스트까지 이루어지며 각 반복주기의 반성 의견을 다음 계획에 포함

애자일 방법론은 다양한 프레임워크를 통해 실천되며, 대표적인 프레임워크로는 **XP(Extreme Programming), Scrum, Kanban**이 있다.

## 2. XP (Extreme Programming)
> 코드 품질과 개발자 협업 방식에 집중한, 기술 실천법 중심의 방법론

XP는 최초의 애자일 프로세스다.
Scrum이 "어떻게 팀을 운영할까"에 집중한다면, XP는 "어떻게 코드를 잘 짤까"에 집중한다.

**XP의 주요 가치 5가지**
- 소통(Cummunication) : 팀 안에서 지식을 공유하고 숨기지 않는다
- 단순함(Simplicity) : 지금 당장 필요한 것만 만든다. 과도한 설계는 낭비다
- 피드백(Feedback) : 테스트, 데모, 코드 리뷰 등으로 빠르게 피드백을 받는다
- 용기(Courage) : 문제가 있는 코드를 과감하게 리팩토링하고, 나쁜 소식을 빨리 공유한다
- 존중(Respect) : 팀원 서로를 존중하고, 코드에 자부심을 갖는다

### 2.1. XP의 핵심 실천법

**① Pair Programming (페어 프로그래밍)**
- 두 사람이 컴퓨터 한 대에서 함께 코드를 작성한다
- 한 명은 코드를 타이핑(Driver), 다른 한 명은 검토·방향 제시(Navigator)
- 역할을 주기적으로 바꾼다
- 실시간 코드 리뷰 효과 + 지식 공유가 자연스럽게 일어난다

**② TDD (Test-Driven Development, 테스트 주도 개발)**
- 코드를 짜기 전에 테스트 코드를 먼저 작성한다
- 실패하는 테스트 → 테스트를 통과하는 코드 작성 → 리팩토링 순서로 반복한다

**③ Continuous Integration (지속적 통합, CI)**
- 팀원 모두가 하루에도 여러 번 코드를 메인 브랜치에 통합한다
- 통합할 때마다 자동으로 빌드·테스트를 돌린다
- 오래 기다렸다가 한 번에 합치면 충돌이 커진다 — 자주 합쳐서 충돌을 작게 유지하자

**④ Refactoring (리팩토링)**
- 기능은 그대로 두고 코드의 구조를 개선하는 것

**⑤ Small Releases (소규모 릴리즈)**
- 작은 단위로 자주 배포한다
- 큰 것을 한 번에 내보내면 피드백이 늦고, 문제가 생겼을 때 원인 찾기가 어렵다

**⑥ Collective Code Ownership (공동 코드 소유)**
- 코드의 특정 부분이 특정 사람의 전담 영역이 아니다
- 누구든 어느 부분이든 수정할 수 있고, 팀 전체가 코드에 책임을 진다
- "그 코드는 A가 짠 거라 나는 몰라요"가 없어야 한다

**⑦ Coding Standards (코딩 컨벤션)**
- 팀 전체가 동일한 코드 스타일을 따른다

## 3. Scrum
> 정해진 짧은 주기(Sprint) 안에 계획-개발-리뷰를 반복하는 프레임워크

XP가 "코드를 이렇게 잘 짜자"라면, Scrum은 "팀을 이렇게 운영하자"다.  
현업에서 가장 많이 쓰이는 방법론이다.

---

### 3.1 Roles

스크럼에는 3가지 역할이 있다.

**Product Owner (PO)**
- 무엇을 만들지 결정하는 사람
- Product Backlog(할 일 목록)를 관리하고 우선순위를 정한다

**Scrum Master (SM)**
- 팀이 스크럼을 잘 할 수 있도록 돕는 사람
- 장애물을 제거하고, 프로세스를 챙긴다
- 관리자가 아니라 **팀을 위한 서번트 리더**에 가깝다

**Development Team**
- 실제로 개발하는 사람들
- self-organizing 팀 : 누가 뭘 할지 스스로 결정한다
- 보통 5~9명이 적정 규모로 알려져 있다

### 3.2 Sprint - 스크럼 주요 구성 요소

> 보통 1~4주 단위의 짧은 개발 주기. 이 안에서 계획-개발-리뷰가 한 사이클로 돌아간다.

- 한 번 시작된 스프린트 중간에 목표를 바꾸지 않는다
- 스프린트가 끝나면 **작동하는 결과물(Increment)**이 반드시 나와야 한다
- 끝나면 바로 다음 스프린트가 시작된다

### 3-3. Event

스프린트 안에서 반드시 해야 하는 4가지 이벤트가 있다.

**① Sprint Planning**
- 이번 스프린트에서 뭘 할지 계획하는 회의
- Product Backlog에서 이번 스프린트에 할 항목을 골라 Sprint Backlog를 만든다
- "이번 2주 동안 뭘 끝낼 수 있을까?"를 팀이 함께 결정

**② Daily Scrum**
- 매일 15분 이내로 진행하는 짧은 미팅
- 세 가지를 공유한다
  - 어제 뭘 했는가
  - 오늘 뭘 할 것인가
  - 막히는 게 있는가 (blocker)

**③ Sprint Review**
- 스프린트 끝에 만든 것을 팀 외부(이해관계자, 고객 등)에 시연하는 자리
- "우리가 이번에 이걸 만들었어요" 를 보여주고 피드백을 받는다
- 문서가 아닌 **실제 작동하는 소프트웨어**를 보여줘야 한다

**④ Sprint Retrospective**
- 팀 내부에서 "우리가 어떻게 일했는가"를 돌아보는 자리
- Keep / Problem / Try 형식으로 많이 진행한다
  - **Keep**: 잘 됐으니 계속할 것
  - **Problem**: 문제가 됐던 것
  - **Try**: 다음 스프린트에서 시도해볼 것
- 리뷰가 결과물에 대한 피드백이라면, 회고는 **프로세스에 대한 피드백**이다

### 3.4 Artifacts

**Product Backlog**
- 제품에 필요한 모든 기능, 개선사항, 버그 수정 등의 목록
- PO가 관리하며 우선순위를 정한다
- 살아있는 문서 : 언제든지 추가/수정/삭제된다

**Sprint Backlog**
- 현재 스프린트에서 개발팀이 완수하기로 약속한 Backlog 항목 목록
- Sprint Planning에서 Product Backlog에서 선택해서 만든다

**Increment**
- 스프린트가 끝났을 때 나온 결과물
- 반드시 **작동하는 상태**여야 한다
- 이전 Increment에 이번 스프린트 결과물이 더해진 것

## 4. Kanban

> 작업을 시각적으로 관리하고, 한 번에 처리하는 작업 수를 제한해서 흐름을 최적화하는 방법론

스크럼이 "정해진 기간 안에 묶어서 처리"라면, 칸반은 "끊임없이 흘러가는 흐름"에 가깝다.

### 4.1. Kanban Board

> 작업의 상태를 Column으로 나눠서 시각한 보드
<img width="800" height="600" alt="image" src="https://github.com/user-attachments/assets/6843843c-1040-4340-8e12-a943129d10ed" />

- 각 작업(카드)는 왼쪽에서 오른쪽으로 이동한다
- 어느 단계에 병목이 생기는지 한눈에 보인다
- GitHub Projects, Jira, Trello, Notion 보드가 이 방식이다

### 4.2 WIP 제한 (Work In Progress Limit)
> 한 Column에 동시에 존재할 수 있는 작업 수를 제한하는 규칙

- WIP 제한을 두지 않으면 → In Progress에 작업이 쌓이고 → 아무것도 완료되지 않는다
- WIP를 제한하면 → 기존 작업을 끝내야 새 작업을 시작할 수 있다 → 흐름이 빨라진다
- 일반적으로 WIP는 `팀 인원수 + 1 또는 2` 혹은 `팀 인원수와 같게` 설정하는 것이 대표적이다
